## A~E 모듈 요약 (뭘 하는지, 어떤 성격인지)

| 모듈 | 뭘 하나 | 어떻게(현재) | 성격 |
|---|---|---|---|
| **[A]** 선순위 보증금 분포 | 나보다 먼저 들어온 N-1세대가 걸어둔 보증금 총액을 확률분포로 추정 | 지역 실거래가에 KDE 적합 → 전세/월세 확률 샘플링 → 몬테카를로 1만회 합산 | 통계적 추정(비지도). **한계**: 면적·연식으로 조건화 안 함 → 학습모델(quantile regression) 업그레이드 논의 중 |
| **[B]** 경매 배당 시뮬레이션 | 경매로 넘어갔을 때 내 보증금 실제 회수액 분포 | 낙찰가율 샘플링 → 낙찰가에서 경매비용·최우선변제금(법정기준표)·근저당·A모듈 선순위 순서대로 차감 | 규칙 기반 시뮬레이션 (법적 배당순위를 코드화, 실제 법령 수치 사용) |
| **[C]** 사고 확률 예측 | 이 건물에서 보증금을 못 돌려받는 사고가 날 확률 | 실질부채비율을 안선영·이상엽(2025) 논문의 실제 로지스틱회귀 오즈비(45만건 데이터 기반)에 대입 → HUG 전국 사고율로 캘리브레이션 | 우리가 학습시킨 모델 아님 — **남이 실제 데이터로 학습시킨 계수를 재사용**. HUG 명단 127명뿐이라 자체 학습셋 구축 불가했던 게 이유 |
| **[D]** 기대손실 | 등급이 아니라 원 단위 금액으로 위험 표현 | `P(사고, C) × (내 보증금 − 예상회수액, B)` | 단순 수식 |
| **[E]** KB 금융상품 액션 | 기대손실과 HUG 보증료를 비교해 가입 여부 판단 | 단순 비용-편익 비교 | 단순 규칙 |

**중요한 구분**: A모듈이 예측하는 건 "선순위 보증금"이 아니라 그냥 "세대별 보증금"이다. "선순위"라는 딱지는 몬테카를로 조립 단계(N-1번 샘플링)에서 우리가 붙이는 것 — 데이터에 "이게 선순위 계약이다"라는 라벨 자체가 없기 때문. 발표 문구에서 "선순위 보증금 예측 모델"이라고 하면 부정확하고, "세대별 보증금 예측 + 몬테카를로 조립"이 정확한 표현.

**AI Challenge 관점에서 지금 상태**: 학습된 모델은 아직 0개(A=통계적 추정, B=규칙, C=남의 계수 재사용, D/E=수식). 그래도 KDE·몬테카를로·실제 논문 계수 재사용 자체가 "그냥 룰"보다는 나은 확률적 모델링이라 직방류(데이터조회+룰+LLM요약)보다는 이미 앞서 있음. 다만 심사에서 "학습된 모델 어디 있냐"는 질문에 확실히 답하려면 A모듈을 실제 지도학습(quantile regression 등)으로 업그레이드하는 게 다음 투자 대상 — 논의 중.

# A모듈 재구축 (AI Challenge 대응 — 학습모델 도입)

사용자 결정: 모델 비교는 원래 계획한 7개 전부(Linear QR, LightGBM, XGBoost, CatBoost, QRF, NGBoost, MDN), 제출까지 시간 여유 있음. Phase 게이팅 방식(각 Phase 끝나면 보고 → 승인 → 다음)으로 진행.

## Phase 0. 데이터 수집 — ✅ 완료

### 한 일
- 수원시 4개 구(장안·권선·팔달·영통) × 최근 36개월(2023.08~2026.07) 전월세 실거래가 API 전량 수집
- `data/raw/rent_deals.csv`로 저장 (재호출 없이 반복 실험 가능)
- 스크립트: `training/phase0_collect_data.py`

### 발견/결과
- **총 49,112건** (중단 기준 1,000건을 훨씬 상회)
- 연도별: 2023(6,908) / 2024(16,631) / 2025(17,075) / 2026(8,498, ~7개월치)
- 구별: 영통구 16,287 / 장안구 12,735 / 권선구 11,474 / 팔달구 8,616
- 주택유형: 다가구 35,183 / 단독 13,929
- 법정동(umdNm) 고유값 55개

### 결정 사항과 근거
- 표본이 충분해 경기도 전체로 확대할 필요 없음 → 수원시 4개구로 범위 확정
- 3년치 확보돼 있어 Phase 1에서 시계열 추세 분석(연도별 트렌드) 가능

### 다음 Phase로 넘어가도 되는가?
- 문제 없음. Phase 1(EDA) 진행 가능.

## Phase 1. EDA — ✅ 완료

### 한 일
- `training/phase1_eda.py` 작성·실행: 기본 프로파일링, 타겟 분포, 이상치, 피처 관계, 카디널리티 5개 섹션 전부 분석
- 그래프 3개 저장 (`docs/eda_plots/`), 상세 결과 문서화 (`docs/EDA_결과.md`)

### 발견/결과
- 중복행 959건(1.95%), 보증금 0원 106건 — 데이터 오류로 판단
- **타겟(보증금) 로그변환 시 skew 2.07→0.005, kurtosis 5.33→-0.145 — 정규분포에 가까워짐**
- **전세(평균 11,827만) vs 월세(평균 1,731만) — 6.8배 차이, 완전히 다른 분포**
- 원본 분포가 이봉(bimodal)인데, **전세/월세로 나누면 해소됨 → 잠재 다봉분포 아님 → MDN 불필요로 결론**
- 면적 상관 0.33, 건축연도 상관 0.27, 시간추세 상관 -0.08(약함), 법정동별 편차 큼, 피처간 다중공선성 없음(전부 |r|<0.13)
- 법정동 55개 중 4개(입북동·중동·영동·상광교동) 표본 10건 미만
- **층(floor) 컬럼은 원본 데이터에 없음 확인 — 지난번 우려가 사실로 확인됨**

### 결정 사항과 근거
- 중복행·0원 제거, 나머지 이상치는 유지(로그변환으로 이미 정규화됨 확인했으므로 IQR/z-score 컷 안 함)
- 타겟은 log1p 변환해서 학습
- 시간 추세는 약하지만 그래도 시간 기반 분할 채택(실제 서비스 상황과 일치, 랜덤분할의 낙관적 편향 방지)
- 희소 동 4개는 구 단위로 shrinkage/fallback
- 층(floor) 피처는 Phase 3 피처 목록에서 제외 확정
- MDN은 모델 비교 후보에서 제외(EDA에서 잠재 다봉분포 미확인) — 나머지 6개(Linear QR, LightGBM, XGBoost, CatBoost, QRF, NGBoost)는 그대로 진행

### 다음 Phase로 넘어가도 되는가?
- 문제 없음. Phase 2(데이터 분할) 진행 가능.

## Phase 2. 데이터 분할 — ✅ 완료

### 한 일
- `training/phase2_split.py`: 중복행·0원 제거 → 시간 기반 분할(train 60% / val 20% / test 20%) → `data/processed/{train,val,test}.csv` 저장
- 상세 결과: `docs/데이터분할_결과.md`

### 발견/결과
- 전처리 후 48,047건 (원본 49,112 - 중복 959 - 0원 106)
- train 28,828 / val 9,609 / test 9,610 (정확히 60/20/20)
- train 기간 2023.08~2025.05, val 2025.05~2025.12, test 2025.12~2026.07
- **전세 비율이 train 30.7% → val 22.0% → test 21.2%로 시간에 따라 뚜렷하게 감소** (전세의 월세 전환이라는 실제 흐름 반영) — 이에 따라 평균 보증금도 4,865만→3,821만으로 감소

# 리포트 개선: 참고 시세 추가

### 배경
- 사용자 지적: 월세 계약의 경우 월세 액수 자체가 보증금과 트레이드오프 관계(전월세 전환율)인데, 지금 모델은 `is_jeonse` 이진값만 쓰고 월세 액수는 피처로 안 씀. 확인해보니 월세 내에서 monthlyRent vs log(deposit) 상관 +0.175로 실제 신호가 있음 — **아직 미반영, 향후 개선 후보로 기록만 해둠** (모델에 넣으려면 시뮬레이션 단계에서 월세액도 면적처럼 별도 분포에서 샘플링해야 해서 추가 작업 필요).
- 사용자 제안: A모듈 출력(숨은 선순위 보증금 총액)만 덩그러니 나오면 사용자가 "이 숫자가 말이 되는지" 검증할 기준이 없음 → 지역 실제 시세를 참고용으로 같이 보여주자.
- 검토 결과: "아예 시세 직접조회로 대체"는 안 됨 — A모듈이 풀어야 하는 문제(건물 전체 누적 선순위 총액)와 시세(세대 1개 통계)는 다른 질문이고, Phase 4에서 이미 "직접조회형 베이스라인(동×면적 중앙값)"이 학습모델보다 못하다는 걸 확인함. 대신 **참고용 보조 정보로만 추가**하기로 함.

### 한 일

# 실제 등기부등본으로 --mortgage 값 교차검증 (2026-07-15)

### 배경
- 근저당(`--mortgage`) 자동조회가 왜 안 되는지 논의하던 중, 확정일자와 달리 **등기부등본 열람 자체는 법적 제한이 없음**(대법원 인터넷등기소에서 누구나 열람 700원/발급 1,000원, 임대인 동의·이해관계 증명 불필요)을 확인. CODEF/대법원 자체 Open API 모두 사업자등록·별도 승인이 필요해 자동화는 여전히 안 되지만, 수동 열람 자체엔 장벽이 없다는 점을 명확히 함.
- 사용자가 테스트 기본 주소(경기도 수원시 팔달구 우만동 39-7)의 실제 등기사항전부증명서(토지·건물 각 1부)를 직접 발급받아 확인.

### 확인된 사실 (PII 제외, 사실관계만)
- **토지·건물 모두 을구(소유권 이외의 권리) "기록사항 없음"** — 근저당권·전세권·가압류·경매개시결정 등 아무 권리도 설정된 적 없음 → 실제 근저당 = 0원. 그동안 테스트에 써온 `--mortgage 0`이 실제 값과 정확히 일치함을 확인.
- 건물 표제부: "연와조 슬래브지붕 3층주택(9가구)", 지층 28.32㎡ / 1층 140.22㎡ / 2층 140.22㎡ / 3층 108.15㎡ — 건축물대장 API로 받아온 `fmlyCnt=9`, 층별개요 fallback 면적(`areas_sqm`)과 **완전히 일치**. 건축물대장 fallback 로직의 정확성이 실물 등기부로 교차검증됨.

# 진단 프레이밍 외부 피드백 검토 + B모듈 조세채권·시나리오 회수율 반영 (2026-07-15)

### 배경
사용자가 외부(다른 세션/조언자)에서 받은 피드백을 공유 — "권리위험/가격위험/임대인이상신호/회수위험" 4축으로 진단을 재구성하고, "전세사기 확률"이라는 표현을 "보증금 미회수 위험도"로 순화하고, 시나리오별 회수율(보수적/기준/낙관적)로 출력을 다층화하라는 내용. 코드·문서를 실제로 대조해서 무엇이 이미 반영돼 있고 무엇이 진짜 갭인지 정리함.

### 검토 결과
- **이미 되어 있던 것**: "사고 확률"(`accident_probability`) 표현은 애초부터 "전세사기 확률"이라고 부른 적이 없어 피드백이 우려하는 문제(검증되지 않은 확률을 단정)는 해당 없음. 전세가율(`jeonse_ratio`, module_c)과 회수위험(Module B 자체)도 이미 구현돼 있었음.
- **진짜 갭으로 확인된 것**: (1) 조세채권(국세·지방세 체납, 특히 당해세는 근저당보다도 우선순위가 높을 수 있음)이 회수 공식에서 빠져 있었음. (2) 결과를 단일 평균/3분류(전액·일부·전액손실)로만 보여줘서 "보수적/기준/낙관적" 시나리오식 표현이 없었음.
- **구현 보류로 정리한 것**: "동일 임대인 다수 주택 보유·반복적 소유권이전 패턴"은 등기부 대량조회가 안 되는(CODEF 사업자등록 요구) 동일한 벽 때문에 사실상 구현 불가로 결론. "권리관계(압류·가압류·신탁) 자동 추출"은 LLM이 등기부 PDF를 직접 읽어 구조화 추출하는 별도의 큰 기능으로 분리 — 이번 라운드에서는 보류하고 조세채권·시나리오 회수율 2가지만 우선 구현하기로 함.

### 한 일
1. **조세채권 반영**: `src/module_b_auction_sim.py`의 `simulate_recovery()`에 `tax_arrears_won` 파라미터 추가, 회수 공식(`distributable = winning_bid - auction_cost - tax_arrears_won - max_repayment_total - mortgage_won - priority_won`)에서 최우선 차감. `report.py`에 `--tax-arrears-won` CLI 인자 추가(기본값 0), 텍스트/JSON 출력·"한계" 문구에 반영. 이 정보도 확정일자와 동일하게 주택임대차보호법 제3조의7상 임대인 동의가 있어야 계약 시점에만 확인 가능해 자동조회 불가 — 수동 입력만 지원.
2. **시나리오별 회수율**: `module_b_auction_sim.summarize()`에 10,000회 몬테카를로 결과의 p10/p50/p90을 `conservative_recovery_won`/`base_recovery_won`/`optimistic_recovery_won`(및 보증금 대비 퍼센트)으로 추가. 리포트 텍스트에 "보수적 X% / 기준 Y% / 낙관적 Z%" 한 줄로 노출.

### 검증
- 조세채권 0원(기본) vs 3,000만원 입력 비교: 회수 예상 68만원→28만원, 전액회수 12.8%→5.2%로 하락, "임대인 국세·지방세 체납 3,000만원을 우선 차감해 반영" 안내 문구 정상 출력.
- 시나리오 회수율: 500만원 소액 보증금 케이스(우만동 39-7, 사고확률 85%대 전액손실 비중 높음)에서 보수적/기준 0%, 낙관적 100%로 출력 — 실제 분포 모양(대부분 전액손실, 일부만 전액회수)과 일치. 5,000만원 구분등기(아파트) 케이스에서는 보수적/기준/낙관적 전부 100%로 정상 출력(근저당 낮고 A모듈 자체가 미적용).
- 다가구·구분등기 두 경로 모두 회귀 테스트 통과, JSON 출력에 새 필드 정상 포함 확인.
